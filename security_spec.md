# Security Specification

## Data Invariants
1. A user profile (`users`) must exactly match `request.auth.uid`.
2. A `jovem` (Youth) belongs to a specific user (the leader). `userId` must equal `request.auth.uid`.
3. An `evento` (Event) belongs to a specific user. `userId` must equal `request.auth.uid`.
4. A `material` (Study Material) belongs to a specific user. `userId` must equal `request.auth.uid`.
5. Timestamps like `createdAt` must be generated by the server.

## The "Dirty Dozen" Payloads
1. **Shadow Update Test**: Payload with `{ isVerified: true }` missing from schema.
2. **Email Spoofing Test**: User creation with unverified email pretending to be admin.
3. **ID Poisoning**: Document creation with a 2000-character string ID.
4. **Relational Sync Exploit**: Creating a `jovem` for another user's ID.
5. **Array Guarding Exploit**: Large array of history logs causing size limit bypass.
6. **Value Poisoning**: Passing a number for `score` instead of an allowed enum string.
7. **Temporal Exploit**: Creating a `jovem` with `createdAt` in the past.
8. **Immortal Field Update**: Attempting to change `ownerId` on an existing `jovem`.
9. **Outcome Reversal**: Modifying a completed event back to an uncompleted state without admin rights (if applicable).
10. **Unauthenticated Read**: Attempting to read a `jovem` collection while signed out.
11. **Cross-Tenant Read**: Attempting to read another user's `jovem` collection.
12. **PII Blanket Read**: Scraping all `jovens` through list queries without matching `userId`.
